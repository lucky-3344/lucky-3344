## 🔑 关键概念

### 区域属性分类（6种）

1. **密集市区** - 商业中心，POI ≥ 30/km²
   - 示例：北京三里屯、深圳华强北
   - 特征：高度城市化，繁华商业

2. **市区** - 城市主要区域，POI 15-30/km²
   - 示例：北京海淀、深圳南山
   - 特征：较为繁华，公共设施完善

3. **县城城区** - 县级中心，POI 8-15/km²
   - 示例：县城中心商业区
   - 特征：中等规模，基础设施完整

4. **乡镇** - 乡镇行政中心，POI 3-8/km²
   - 示例：浙江乌镇、江南水乡
   - 特征：小规模城镇，人口中等

5. **农村** - 村落地区，POI < 3/km²
   - 示例：边远山区、农业县
   - 特征：稀疏分布，主要为农业

6. **郊区** - 开发区/工业区
   - 示例：经济开发区、工业园
   - 特征：特殊用途，POI分布不规律

### 坐标系统

- **WGS84** - GPS标准坐标系，全球通用
- **GCJ02** - 中国坐标系，高德地图专用
- **转换误差** - 1-2米（正常范围）

### API调用

- **逆地理编码** - 获取坐标对应的地址
- **周边查询** - 查询坐标周边的POI
- **地理编码** - 获取地址对应的坐标

## 💡 使用建议

### ✅ 最佳实践

1. **数据准备**
   - 使用完整地址（市-区-街道）
   - 验证坐标格式（经,纬）
   - 保留原始数据备份

2. **API管理**
   - 定期更新API密钥
   - 监控调用额度
   - 处理超时异常

3. **结果验证**
   - 对异常结果进行手动检查
   - 结合行政区划信息判断
   - 了解POI数据的局限性

### ❌ 常见误区

1. ❌ 混淆坐标系统（WGS84 vs GCJ02）
   ✓ 程序已自动处理转换

2. ❌ 忽视POI数据滞后性
   ✓ 参考返回的行政区划信息

3. ❌ 一次性处理超大数据量
   ✓ 考虑分批或使用线程池

4. ❌ 使用过期的API密钥
   ✓ 定期检查和更新
